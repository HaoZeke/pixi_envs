* Purpose

Originally: candidate inputs for a pair of defensive guards proposed
upstream. After tracing showed =NBinStandard::setup_bins= already floors
=nbinx/y/z= to 1 and errors on box << cutoff, those guards were
retracted (see verdict section in
=~/Git/Gitlab/obsidian-notes/Software/Metatomic/KokkosNL_StockVsMetatomic_Divergence.org=).

Remaining upstream fix:
=HaoZeke/lammps:fix/kokkos-package-nbin-atoms-per-bin-validation= --
reject zero / negative values for =package kokkos nbin/atoms/per/bin=
at the parser in =src/KOKKOS/kokkos.cpp=. The =in.atoms_per_bin_zero=
input below is the demonstration: pre-patch it runs to completion
(auto-grow in =NBinKokkos::bin_atoms()= masks the invalid value);
post-patch it aborts at input parse with /"Illegal package kokkos
nbin/atoms/per/bin value: must be > 0"/.

=in.mbins_zero= is kept as historical record of an investigation that
concluded the state it tries to produce is not reachable on stock
LAMMPS.

Context: metatensor/lammps#45 review comment r2954700561 (Luthaf, "Should we
send this upstream directly?"). The downstream PR carries these guards
mixed with metatomic-specific logic; only the two general-purpose guards
belong upstream.

* Status :: stock LAMMPS cannot cheaply trigger the SIGFPE

Empirical testing on rg.cosmolab (patch_10Sep2025-734-ga1c7f5baba,
Kokkos CUDA, RTX 4070 Ti SUPER) with the two inputs below:

| Input                   | Invocation                               | Result                                  |
|-------------------------+------------------------------------------+-----------------------------------------|
| =in.atoms_per_bin_zero= | =lmp -k on g 1 -sf kk -in ...=           | Completes normally, no crash            |
| =in.mbins_zero=         | =lmp -k on g 1 -sf kk -in ...=           | Completes normally, no crash            |

Root cause: =NBinKokkos::bin_atoms()= at =src/KOKKOS/nbin_kokkos.cpp:121=
auto-grows =atoms_per_bin += 16= on overflow, so a user-supplied zero gets
promoted to 16 on the first bin-atoms pass. Similarly, large =neigh_modify
binsize= values larger than the shortest box dim are handled gracefully
in =NBinStandard::setup_bins=.

Conclusion: under normal stock-LAMMPS configurations these guards do not
fire. They remain valuable because:

1. =pair_metatomic= in metatensor/lammps#45 reached these paths and
   triggered real SIGFPEs in =NPairKokkosBuildFunctorGhost= on PET-MAD-like
   dense-system + large-cutoff setups.
2. Other custom pair styles with their own NL estimation or that bypass
   the auto-growth logic can reach the same degenerate states.
3. The guards themselves are a single integer comparison each --
   essentially free at runtime and trade a SIGFPE for a clear
   =ERROR: Kokkos neighbor list ...= message.

* Running

Assuming a Kokkos-enabled =lmp= is on =PATH=. On rg.cosmolab use
=/home/goswami/conda/envs/lmp-mta/bin/lmp= (Kokkos CUDA + OpenMP).

#+begin_src sh
# From this directory:
lmp -k on g 1 -sf kk -in in.atoms_per_bin_zero
lmp -k on g 1 -sf kk -in in.mbins_zero

# Or via pixi tasks from orgs/metatensor/lammps/:
pixi run repro-atoms-per-bin   # will complete without error (see Status)
pixi run repro-mbins           # will complete without error (see Status)
#+end_src

Neither reproducer currently produces the crash on stock upstream develop;
they are kept as historical record of what was tried and as smoke tests
that the guard code paths do not regress under normal operation.

#+begin_src sh

cd /home/goswami/pixi_envs/orgs/metatensor/lammps/lammps   # or a fresh clone

# Rebuild LAMMPS with -D PKG_KOKKOS=on -D Kokkos_ENABLE_CUDA=on.

Because the two guards only add defensive =if= checks, normal inputs
(including the two reproducer files) should behave identically on the
patched and unpatched builds.
